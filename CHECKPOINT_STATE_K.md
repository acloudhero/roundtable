## Scope and rationale

Checkpoint K combines the originally-separate K, L, and M scopes from
`docs/PWA_READINESS.md` § 17 because they form a single cohesive
shippable unit:

- K alone (manifest + icons + HTML head) is not installable without
  the service worker (L).
- L alone (SW + update banner) is not installable without the
  manifest (K).
- M alone (mobile CSS) delivers no PWA capability but is needed for
  the installed app to feel right.

Shipping K+L+M together lets a single deployment go from "v0.12.0
RC" to "installable PWA in Chrome / Edge / Brave / Safari / Android
Chrome." Hosted validation is the natural next step (NOT part of
this checkpoint per the brief).

## Service worker and update banner

**Generated SW:** `dist/sw.js` (2055 bytes minified) + `dist/workbox-2fbc6a65.js`
(15026 bytes Workbox runtime). Precaches 18 entries (~507 KiB) covering
the full app shell.

**Strategy:** Precache the app shell, fall through to network for any
non-precached request (which never happens because the app makes
zero runtime network calls). `navigateFallback: 'index.html'` lets
deep-link reloads work offline.

**Update flow** (`registerType: 'prompt'`):

1. SW detects new bundle on page load.
2. Downloads new SW + precache in background.
3. New SW enters `installed` waiting state.
4. `useRegisterSW`'s `needRefresh` flips to `true`.
5. `PwaUpdateBanner` renders the amber "Update Available" banner
   above `StoragePressureBanner`.
6. Operator clicks **Reload to update** → `updateServiceWorker(true)`
   posts `SKIP_WAITING` to the SW; the SW activates; the page
   reloads.
7. Operator clicks **Later** → banner dismisses for the session;
   SW continues waiting; banner reappears on the next page load.

**Offline-ready one-shot** (first install):

1. SW completes initial precache.
2. `useRegisterSW`'s `offlineReady` flips to `true`.
3. `PwaUpdateBanner` renders the green "Offline Ready" banner.
4. Auto-dismisses after 8 seconds; also manually dismissible.

## Storage persistence request

Called inside `useRegisterSW`'s `onRegisteredSW` callback in
`src/components/PwaUpdateBanner.tsx`:

```ts
if (typeof navigator !== 'undefined' &&
    navigator.storage &&
    typeof navigator.storage.persist === 'function') {
  void navigator.storage.persist().then(
    (granted) => { /* console.info log */ },
    () => { /* swallow */ }
  );
}
```

Behavior:

- Best-effort, non-blocking, fire-and-forget.
- Triple guard against missing API surface (cross-browser safety).
- Logs result to `console.info` for operator inspection in DevTools.
- Never throws; never retries; never surfaces UI on failure.
- Granted → browser commits to not evicting RoundTable's
  localStorage without warning. Denied → browser's default eviction
  policy applies (LRU-ish).

## Mobile UX hardening

All CSS-only. Appended to `src/styles/app.css`:

| Concern | Rule |
|---|---|
| iOS notches / dynamic island | `.app-header { padding-top: max(0px, env(safe-area-inset-top)); }` |
| iOS home indicator | `.tab-nav { padding-bottom: max(0px, env(safe-area-inset-bottom)); }` |
| Landscape on notched devices | `.app-shell { padding-left: env(safe-area-inset-left); padding-right: env(safe-area-inset-right); }` |
| Viewport-fit | `<meta name="viewport"... viewport-fit=cover>` (in index.html) |
| Pull-to-refresh | `html, body { overscroll-behavior: contain; }` |
| iOS tap-highlight flash | `a, button, [role=button], .tab-btn, .btn { -webkit-tap-highlight-color: transparent; }` |
| Tab selection menu | `.tab-nav, .tab-btn { user-select: none; -webkit-user-select: none; }` |
| Touch-target floor | `.btn-primary, .btn-secondary, .btn-danger, .btn-danger-solid { min-height: 44px; }` plus `.tab-btn { min-height: max(44px, var(--tab-height)); }` |

Notes:

- `overscroll-behavior: contain` (not `none`) preserves bounce
  within nested scrollers while preventing the document-level
  pull-to-refresh gesture.
- Touch-target floor applies to *primary* buttons. Dense list
  buttons (Raw Notes Delete, banner action buttons) retain their
  compact 28-32 px sizing via inline-style overrides at the call
  sites — these are not the primary affordances for any workflow,
  so the trade-off is intentional.
- No new theme tokens; no layout architecture changes; no
  redesigns.

## Known limitations

Carried from `docs/PWA_READINESS.md` plus new ones surfaced during
Checkpoint K:

1. **Hosted Netlify validation has not been performed.** All
   build-time and local-preview verifications pass, but the live
   Netlify deployment has not been smoke-tested. This is the
   natural next non-checkpoint task — see `docs/RELEASE_CHECKLIST.md`'s
   new "v0.12.0 PWA" section for the manual verification list.
2. **No focus trap on Checkpoint J modals.** Carried from J;
   unchanged by K.
3. **No splash screens beyond manifest `background_color`.**
   Operators on iOS see a `#0d0f11` background before the app
   loads — no logo. Per-device splash PNGs require a long list
   of `apple-touch-startup-image` link tags; deferred to v1.1.
4. **No periodic update check while the PWA is open.** The SW
   checks for new versions on page load. An operator who keeps
   the PWA open for hours won't see updates until they reload
   manually. Could be added with a periodic
   `registration.update()` call but kept out of scope to ship clean.
5. **localStorage quota still ~5 MB on iOS Safari.** IndexedDB
   upgrade is Checkpoint O, deferred. The new
   `navigator.storage.persist()` call helps with eviction but
   does not raise the quota.
6. **`navigator.storage.persist()` not supported on iOS Safari**
   (as of WebKit 18). The safety guard short-circuits cleanly;
   no UI impact; no error logged. Operators on iOS rely on the
   browser's default durability for installed PWAs.
7. **iOS Share Sheet flow** for downloads when the PWA is
   installed — documented in `docs/PWA.md` and
   `docs/MARKDOWN_HANDOFF.md`. Not a regression; how iOS handles
   `<a download>` in standalone mode.
8. **No in-app "Install RoundTable" affordance.** The
   browser-provided install affordance (URL bar icon on Chrome /
   Edge / Brave; Share Sheet on iOS Safari) is sufficient.
   Custom prompt UI is a v2 consideration.
