# v0.11.0 Markdown Handoff Mode — Feasibility Assessment & Phase Plan

Reviewed against the v0.10.5 baseline at `/home/claude/rt-v105/`. The short answer: this is implementable as a feature release without a rewrite, but it is a *substantial* release — larger in surface area than v0.10.3/v0.10.4/v0.10.5 combined — and the boundary between "must-have for v0.11.0" and "deferred to v0.11.x" matters more than usual. Below, the 14 answers, in order, with the architectural reasoning and the decisions I want locked before any code is written.

## 1. Feasibility — yes, as a feature release

Markdown Handoff Mode is a workflow-layer addition, not an architectural change. The v0.10.5 codebase already provides the seams it needs:

- **`AppState` is a single versioned object** with a working migration engine (`src/utils/migration.ts`) and validation firewall (`src/utils/validation.ts`). New top-level arrays can be added cleanly via a 0.10.5→0.11.0 migration step that just defaults them to `[]` — same pattern as Phase 7B's `promptWrappers`.
- **Markdown generation already lives in one place** (`src/utils/markdownExport.ts`) and uses the exported `fence()` helper for dynamic-tilde fencing. Single-source-of-truth artifact functions extend naturally from here.
- **The mediator extractor is already line-based and tolerant** (v0.10.5 rewrite). Adding code-fence awareness is one extra state machine variable, not a rewrite.
- **The functional updater pattern** (`updateRoundFunctional`, `AppStateUpdater`) gives safe composition for the import-and-snapshot transactions that rollback requires.
- **Validation/migration already handles structured imports** and produces grouped warnings. Reusing this for `.md` artifact imports avoids inventing a parallel system.

What does *not* exist and must be built: YAML frontmatter generation/parsing, browser SHA-256 hashing, a Raw Notes substrate, an Import Transaction / rollback substrate, file-system download/upload UI affordances, and a code-fence-aware extension to the structured-synthesis parser. None of these require rewriting; all extend cleanly.

**Honest caveat:** the v0.11.0 surface area approaches the upper limit of "feature release." If scope expands further (say, in-app .md viewer rendering, stitching UI, or bulk operations), it crosses into Phase 8 territory. The discipline of the must-have/deferred split in Q11 is what keeps this a release, not a rewrite.

## 4. Markdown export structure and YAML frontmatter

Every artifact:

```
---
artifact_type: roundtable.markdown.v1
source_kind: mediator_packet
schema_version: 0.11.0
app_version: 0.11.0
artifact_id: art_20260519T153000Z_a1b2c3
exported_at: 2026-05-19T15:30:00Z
project_id: proj_abc123
project_name: RoundTable v0.11.0 Planning
round_id: round_007
round_number: 7
canonical_state_hash: sha256:4f3a...
prompt_hash: ~
content_hash: sha256:8c2e...
generated_by: roundtable
---

# Mediator Packet — Round 7
[body]
```

Rules:

- The frontmatter must be the **first bytes** of the file. Opening `---` on line 1, closing `---` on its own line. Anything not matching this shape is treated as a no-frontmatter file → routed to Raw Notes on import.
- `content_hash` covers the **body only**, after normalization (Q5).
- Fields not applicable to a given `source_kind` are emitted as YAML null (`~`) rather than omitted — keeps the schema flat and the parser simple.
- `artifact_type` is the namespace + major version. `roundtable.markdown.v2` would be a breaking format change.
- The body uses the existing dynamic-tilde fencing for code blocks (the `fence()` helper from v0.10.3).

## 5. Normalization and hashing — exact spec

The brief asks for a deterministic strategy. Lock these now:

**Normalization (applied before hashing the body):**

1. Decode input as UTF-8.
2. Strip a leading BOM (`U+FEFF`) if present.
3. Apply Unicode NFC normalization.
4. Replace all `\r\n` and standalone `\r` with `\n`.
5. Ensure exactly one `\n` at end-of-string: strip trailing `\n` runs to none, then append exactly one.
6. **Do not touch leading or trailing whitespace on individual lines.** This preserves intentional whitespace inside code fences, indented lists, and ASCII tables.

That last point is a deliberate tradeoff. Stripping trailing whitespace on every line would be cleaner for diffs, but it's destructive inside code blocks. False stale-state warnings caused by *editor* whitespace mutation are documented as a known limitation; the mitigation is the import-preview workflow.

**Hashing:**

- SHA-256 via `crypto.subtle.digest('SHA-256', encodedBody)`.
- Output: lowercase hex.
- Format in frontmatter: `sha256:hexstring`. The `sha256:` prefix is a namespace that allows a future algorithm change without breaking the schema.

**What is hashed for what:**

- `content_hash` — the artifact's body, post-normalization.
- `canonical_state_hash` — the project's `canonicalState` string, post-normalization.
- `prompt_hash` — the `promptText` of the GeneratedPrompt that produced the response, post-normalization. Only populated for `source_kind: model_response`.

**SubtleCrypto availability:** present in all modern browsers, but only in secure contexts (HTTPS or localhost). RoundTable already runs on localhost in dev and is local-first; production deployment over `file://` is *not* supported (SubtleCrypto unavailable). Document this; it's a real constraint.

## 6. Import, validation, conflict resolution, stale-state, rollback, Raw Notes

The import pipeline is a state machine with one preview gate and one commit gate. No imported data may bypass either.

**Pipeline:**

1. User uploads `.md` (or pastes text into the existing import textarea — both feed the same path).
2. **Frontmatter split.** If the file doesn't start with `---\n...\n---\n`, treat as no-frontmatter → preview-as-Raw-Notes.
3. **YAML parse.** Failures → preview-as-Raw-Notes with the YAML error in warnings.
4. **Schema validation** (handwritten guards by `source_kind`). Failures classified by which field is wrong.
5. **Body normalize + content hash recompute.** Compare to `frontmatter.content_hash` → detect post-export manual edits.
6. **Lookup** `project_id`, `round_id`, prior `artifact_id`.
7. **Current-state hashes** computed for relevant slices; compared to frontmatter hashes.
8. **Build `ImportPreview`** with parsed body, target slice, and warnings grouped by severity.
9. **User commits / forks / Raw-Notes / cancels.**
10. **On commit:** snapshot the affected slice into a new `ImportTransaction`, apply the change via `updateRoundFunctional` (so it composes correctly), append to `importHistory`. The commit happens in a single functional dispatch so partial application is impossible.

**Conflict resolution matrix** (the brief explicitly asks for the resolution UX, not just detection):

| Conflict | Severity | Default UX | Available alternatives |
|---|---|---|---|
| `artifact_type` unknown / newer-major | Error | Hard block | "Import as Raw Notes" |
| `source_kind` invalid | Error | Hard block | "Import as Raw Notes" |
| `schema_version` newer than app | Error | Hard block (preserves future-schema rejection) | "Import as Raw Notes" |
| `schema_version` older, migratable | Auto | Run existing migration silently | n/a |
| `project_id` not found | Warning | Show project picker | "Map to project X" / "Import as Raw Notes" |
| `round_id` not found | Warning | Show fork option | "Fork into new round" / "Map to round Y" / "Import as Raw Notes" |
| `canonical_state_hash` mismatch | Warning | Highlight in preview | "Import anyway" / "Show diff (deferred to v0.11.1)" / "Cancel" |
| `prompt_hash` mismatch (on model_response) | Warning | Highlight in preview | "Import anyway (treat as response to current prompt)" / "Cancel" |
| `content_hash` mismatch with frontmatter | Warning | Highlight ("file was edited after export") | "Import anyway" / "Cancel" |
| `content_hash` matches a stored artifact | Info | "Already imported" | "Re-import as new" / "Cancel" |
| Malformed frontmatter | Warning | Default-routes to Raw Notes | (cannot import as authoritative) |
| Unparseable body | Warning | Default-routes to Raw Notes | (cannot import as authoritative) |
| Truncated (missing closing fence, missing required sections) | Warning | Allow commit + flag | "Import as Raw Notes" / "Mark as Part 1 of N (deferred — for now: just save with warning)" |

**Raw Notes** is the universal fallback. Every failure path that can't safely commit produces a `RawNote` with every field that could be parsed, plus the raw body verbatim, plus all warnings. The user can later inspect, copy out content, and manually re-enter into the right place. Raw Notes is *not* re-importable — that's by design. If automatic import failed once, it'll fail again the same way; the user needs to look at it.

**Rollback (v0.11.0 scope):**

- Pre-commit snapshot is part of the `ImportTransaction`. Only the affected slice is snapshotted, not the whole `AppState` (to keep `importHistory` bounded).
- Rollback restores the snapshot via `updateRoundFunctional` against the current `prev`.
- Only the most recent un-rolled-back transaction is rollable. Older transactions are visible in the history with a greyed-out rollback button and an explanation. Cascading rollback (rolling back transaction N when transactions N+1..M came after) is deferred to v0.11.1 because it requires dependency analysis I don't want to ship without thorough testing.
- After rollback, the transaction stays in the history (marked `rolledBackAt`), so the user can see what happened.

## 7. Code-fence-aware synthesis parser

The v0.10.5 line parser (`extractMediatorSections`) currently runs `isHeadingShaped(line)` against every line. The 0.11.0 extension adds one state variable and one check:

```
state    : 'OUTSIDE' | 'INSIDE_FENCE'
fenceTok : '~' | '`' | null       // which char opened the current fence
fenceLen : number                  // length of opening run

for each line:
  trimmed = line.trim()
  if state == OUTSIDE:
    m = trimmed.match(/^(`{3,}|~{3,})/)
    if m:
      state = 'INSIDE_FENCE'
      fenceTok = m[1][0]
      fenceLen = m[1].length
      append to current section body (the fence line is body content)
      continue
    if isHeadingShaped(line):
      flush()
      currentKey = detectMediatorHeading(line)  // may be null for unknown
      continue
    append to current section body

  if state == INSIDE_FENCE:
    // Closing fence: same char, length >= opening
    m = trimmed.match(new RegExp('^' + fenceTok + '{' + fenceLen + ',}$'))
    if m:
      state = 'OUTSIDE'
      fenceTok = null
      fenceLen = 0
    append to current section body  // including the fence line itself

on EOF:
  if state == INSIDE_FENCE:
    emit warning 'UNCLOSED_CODE_FENCE'
    // implicit close: don't reset state; just flush whatever section is open
  flush()
```

Properties:
- Heading detection inside a fenced block is suppressed — `### Heading` inside ``` ``` ``` stays in body.
- A missing closing fence does *not* crash; it just produces a warning and treats the remainder as body of the current section. Combined with the existing "unknown heading flushes to null" rule, this preserves the canonical principle: known content is never erased by malformed input.
- Indented code fences (4-space indent) are intentionally *not* tracked. They're rare in mediator output and adding them complicates the state machine for marginal value. Documented limitation.
- Tilde and backtick fences are both supported. The opening character locks; the closing fence must match.

This parser lives in `src/utils/markdownParse.ts` and is called by both `mediatorExtract.ts` (for structured synthesis import) and any future generic-Markdown consumer.

## 8. Truncation and stitching — design now, defer most implementation

The brief asks for stitching design even if not first to ship. What I'd commit to in v0.11.0:

**Land:**
- The `part?: { index, total }` field is *reserved* in frontmatter (writer doesn't emit; reader recognizes).
- The parser emits a `POTENTIALLY_TRUNCATED` warning when it sees any of: missing required section headings (synthesis only), unclosed code fence, body ending mid-paragraph (heuristic: last non-blank line doesn't end with `.`/`!`/`?`/`)`/`]`/`>`).
- Partial-import results land in Raw Notes with `importStatus: 'partial'` and the warning preserved.
- Cross-references between related raw notes (e.g., two `partial` notes for the same project/round within a short time window) are recorded as a sibling-ids list so the future stitcher can find them quickly.

**Defer to v0.11.1:**
- The actual merge UI (overlap detection, join-point preview, manual override).
- The combined "stitch & commit" action.
- Multi-part frontmatter reconciliation (when Part 1 says `total: 2` and Part 2 says `total: 3`).
- Reordering of out-of-canonical-order sections during stitch.

This means a user in v0.11.0 with truncated mediator output can still import both parts safely (both land in Raw Notes), can still see them clearly, and has the choice to either manually concatenate-and-re-import or wait for v0.11.1. No silent data loss either way.

## 9. UI changes

Additive, no redesign. Concretely:

- **Side-by-side controls.** Each existing "Copy" button gains a "Download .md" sibling. Same source string. Existing "Paste" textareas gain a "Or upload .md" file picker. Both wired to the same import handler.
- **Import Preview modal** — required intermediate surface between file selection and commit. Sections: detected artifact type + target, warnings (Errors / Warnings / Info groups), parsed-content preview (rendered as plain Markdown text in a `<pre>`, no Markdown renderer dependency for v0.11.0), action row (Import / Fork / Raw Notes / Cancel).
- **Raw Notes panel** — new tab in main nav. List view with filters by source kind / project / date. Each row expands to show raw body + parsed frontmatter + warnings. Per-row actions: copy raw body, delete (with confirm).
- **Import History panel** — new tab in main nav. Reverse-chronological. Each row shows source artifact, target, timestamp, change summary, rollback button (enabled only for most recent un-rolled-back). Rollback prompts for confirmation and a reason.
- **Stale-state indicators** — small badges on the Mediator and Round Builder panels when `canonicalStateHashAtGeneration` (on the prompt) doesn't match the project's current canonical state hash. This *replaces* the v0.10.3 60-char prefix heuristic with hash-based detection.
- **Generation provenance** — when a prompt is generated, capture and store `canonicalStateHashAtGeneration`. This needs to be done at generation time, not at export time, so the hash is durable in `GeneratedPrompt` itself.

What we are *not* adding for v0.11.0:
- Markdown render preview (plain text suffices).
- Drag-and-drop file upload (file picker only).
- Bulk export.
- Re-import of Raw Notes.
- An in-app diff viewer (linked from canonical-state-mismatch warning).

## 10. Migration & backward compatibility

**Migration step 0.10.5 → 0.11.0:**

```
- if !state.rawNotes: state.rawNotes = []
- if !state.importHistory: state.importHistory = []
- (no other shape changes)
```

Both defaults are pure structural additions. The existing migration engine emits a `MIGRATION_STEP_APPLIED` notice with the new field paths so the import preview surfaces the defaulting (matching the 0.7→0.8 pattern).

**Existing data:** GeneratedPrompts that pre-date v0.11.0 have no `canonicalStateHashAtGeneration`. Stale-state detection silently skips those prompts (it's an optional field; missing = "we don't know, suppress warning"). New prompts generated in v0.11.0+ get the hash.

**Backward compatibility for the user workflow:**

- All existing Copy buttons keep working. All existing paste textareas keep working. The .md path is purely additive.
- Existing localStorage state (under `roundtable.appState.v1`) loads unchanged. The migration step runs the first time v0.11.0 sees a v0.10.5 state and persists the upgraded shape.
- Future-schema rejection (`UNSUPPORTED_SCHEMA_VERSION`) preserves: v0.11.0 export imported into v0.10.5 → blocked at the validation firewall.

**Downgrade is lossy.** Documented limitation: opening v0.11.0 data in v0.10.5 drops `rawNotes` and `importHistory` on a round-trip. Users who need to downgrade should export their JSON in v0.11.0, downgrade, and accept the loss. This isn't fixable without storing the new fields under a "metadata bag" key that v0.10.5 ignores, and that's over-engineering for a forward-only roadmap.

## 12. Concrete acceptance criteria

These must all pass before v0.11.0 ships:

1. **Same-source guarantee.** For every artifact source kind, automated test confirms that the string passed to Copy is byte-identical to the string written to download. Implementation: artifact generators return `{ fullText }`; both UI paths consume that string.
2. **Round-trip integrity.** Export each of the five artifact source kinds, import in a fresh tab with the same project state, verify zero warnings and zero divergence.
3. **Stale canonical-state detection.** Export a mediator packet, edit the project's canonical state, re-import the packet → user sees `CANONICAL_STATE_HASH_MISMATCH` warning. Import is blocked from being silent.
4. **Stale prompt detection.** Export a model response artifact, regenerate prompts (so prompt hash changes), re-import → `PROMPT_HASH_MISMATCH` warning.
5. **Post-export edit detection.** Export, manually edit the body in a text editor, re-import → `CONTENT_HASH_MISMATCH` warning ("file was edited after export").
6. **Malformed YAML → Raw Notes.** Import a file with `---\nbroken: : : ---\n[body]` → Raw Notes entry with full body preserved and YAML parse error in warnings. No crash.
7. **Truncated body → Raw Notes with partial flag.** Import a packet ending mid-code-fence → Raw Note with `importStatus: 'partial'` and `UNCLOSED_CODE_FENCE` warning.
8. **Code-fence-aware extraction.** Synthesis containing a `### Risks` heading *inside* a fenced code block does not get treated as a section heading — body remains intact in the previous section.
9. **CRLF/LF stability.** Same logical file with `\r\n` vs `\n` line endings → identical `content_hash` → no false stale warnings.
10. **Rollback restores state byte-identically.** Import → snapshot capture → rollback → resulting AppState slice equals the pre-import slice deep-equality.
11. **No silent data loss.** For any input file the parser sees, the union of (committed body) + (Raw Notes body) contains every non-whitespace byte from the input.
12. **Forward-schema rejection preserved.** A v0.11.0 file imported into a build with `SCHEMA_VERSION = '0.10.5'` is rejected with `UNSUPPORTED_SCHEMA_VERSION` — the existing v0.7B / v0.10.x firewall is not regressed.
13. **Migration safety.** Loading an existing v0.10.5 localStorage state into v0.11.0 produces `state.rawNotes = []`, `state.importHistory = []`, no other shape changes, and the migration notice is surfaced if state is exported and re-imported.
14. **No new network surfaces.** `grep -rEn "fetch\(|XMLHttpRequest|axios|api\.|googleapis"` in `src/` returns nothing new.
15. **Existing v0.10.5 workflows unaffected.** All v0.10.5 Copy and paste paths produce identical results to v0.10.5. (Spot-checked via regression scenario from `docs/RELEASE_CHECKLIST.md`.)

## 13. Risks

**Technical:**

1. **SubtleCrypto in non-secure contexts.** Users running RoundTable over `file://` lose hashing — and therefore lose half of Markdown Handoff Mode (stale-state detection breaks; new exports can't compute `content_hash`). Mitigation: degrade gracefully — if SubtleCrypto unavailable, omit hash fields in frontmatter and disable stale-state UI, with a banner pointing to localhost or HTTPS hosting. Document prominently.
2. **YAML edge cases.** Even with js-yaml restricted to a flat schema, weird values (multi-line strings via `|` block, dates auto-cast to `Date`, etc.) can corrupt fields. Mitigation: post-parse, coerce every field through a typed guard that rejects anything that isn't a string/null/number/object-with-known-shape.
3. **Migration corruption.** A buggy 0.10.5→0.11.0 step could destroy state. Mitigation: the step is the simplest possible (defaulting two `[]` arrays). Pre-write a snapshot of pre-migration state to a temp localStorage key for one-version-back recovery.
4. **Code-fence parser edge cases.** Real Markdown has nested fences, indented fences, edge cases in fence-character matching. Mitigation: scope to standard ` ``` ` and `~~~` fences, document indented fences as out of scope, fuzz-test with a corpus of real GPT/Claude/Gemini outputs before release.
5. **localStorage growth.** Raw Notes and Import History can balloon. Mitigation: bounded ring buffers (200 and 50 default), with a "N older entries pruned" banner before discard.

**UX:**

1. **Warning fatigue.** Three severity levels and a dozen warning codes will overwhelm users. Mitigation: default actions are always the safest one (Cancel / Import as Raw Notes); "Import anyway" requires a deliberate two-step click; warnings are grouped and the most severe is shown first.
2. **Rollback ambiguity.** Users may not understand what rollback actually undoes. Mitigation: rollback confirmation modal shows "this will restore: round X's modelResponses, project Y's canonicalState" — explicit list, not just a yes/no.
3. **Raw Notes graveyard.** Users dump failures and forget them. Mitigation: surface count in nav badge; periodic prompt to review/clean up; export-Raw-Notes-as-zip affordance (deferable to v0.11.1).
4. **Two import paths confuse new users.** Paste-text-into-textarea (legacy) vs. upload-.md-file (new). Mitigation: both feed the exact same pipeline; UI labels them as "Upload .md (recommended)" and "Or paste Markdown text". The recommendation steers users toward the .md path without removing the legacy option.

**State-corruption:**

1. **Most-recent-only rollback can confuse cascading workflows.** If the user does Import A → Import B → Decision recorded → Rollback A: Decision still references A's effects. Mitigation: document; surface a warning when rolling back A would orphan B's reliance on A.
2. **Concurrent multi-tab edits** (already a v0.10.x risk) become higher-stakes with imports. Mitigation: document; the existing localStorage last-write-wins behavior is unchanged.
3. **Hash collisions.** Cryptographically infeasible; not a real risk.

## 14. Decisions required before coding

These must be resolved by the reviewer / mediator before implementation begins. I include my recommendation for each:

## Bottom line

v0.11.0 is implementable as a feature release from v0.10.5 without rewriting the app. It is the largest release since Phase 7B and approaches the upper bound of "feature release"; the discipline of the must-have/deferred split is what keeps it from sliding into a Phase 8 rewrite. The critical guarantees — **no silent data loss, no canonical state corruption, malformed content preserved in Raw Notes, copy/paste workflow intact, rollback available, preview/copy/download from the same source string** — are all achievable with the architectural sketches above.

The two riskiest pieces are (a) the hash-based stale-state detection becoming source of UX confusion (false positives from editor whitespace mutations) and (b) the most-recent-only rollback constraint becoming source of UX confusion (users expecting cascading). Both are documented limitations, both are surfaced in the import preview, and both have clear upgrade paths in v0.11.1.

I recommend proceeding with implementation after the 16 Q14 decisions are locked. If even one is unresolved (especially #1, #2, #4, or #10), do not begin coding — those propagate into the type system and changing them later costs disproportionately.
