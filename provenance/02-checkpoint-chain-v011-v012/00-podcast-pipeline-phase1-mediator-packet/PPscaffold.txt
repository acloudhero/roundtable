1. Architecture summary
A single-process, CLI-driven Python pipeline that moves an approved text through deterministic stages (ingest → approve → synthesize → assemble → publish) to produce one MP3 and one RSS item, while emitting internal pipeline events that a thin telemetry adapter maps to OpenTelemetry traces, metrics, and logs.
The architecture rests on five load-bearing decisions:
pipeline_run_id is the durable correlation key, stored in SQLite and carried in every artifact, event, log record, and span attribute. Trace IDs are ephemeral; pipeline_run_id is not.
Stages are plain function calls taking a shared PipelineContext, not an event bus. The event model is data, not transport.
Artifacts on disk are the inter-stage contract, not in-memory objects. This is the single architectural decision that makes future cloud migration mechanical rather than a rewrite.
The telemetry adapter is the only module that imports opentelemetry. All other modules emit framework-neutral internal events; the adapter translates. This preserves the "OTel decoupled through an adapter" constraint and makes the adapter swappable or stubbable.
Manual approval is a real stage with persisted state, not a sleep-loop or a stdin prompt buried inside synthesis. Approvals create linked-trace boundaries because operator wall-clock time can be hours or days.
Everything else is consequence.
2. Proposed repository structure
Code
Two non-obvious choices worth defending:
runs/<pipeline_run_id>/ as a self-contained directory. Mirrors the prefix-per-run pattern in S3/GCS. The cloud migration replaces runs/ with a bucket prefix and nothing else changes about the artifact contract.
feed/ separate from runs/. Published artifacts are immutable once in the feed. Runs are working state. Mixing them produces deletion accidents.
3. Python package/module structure
Module-level responsibilities, with import direction rules (lower may not import upper):
Code
The hard rule that prevents tangling: events imports nothing internal, and adapters/telemetry is the only place where events meets opentelemetry. Any stage that imports opentelemetry directly is a bug.
I'd recommend enforcing the import direction with import-linter from day one. The cost is 20 lines of config; the savings are real.

Design notes:
Every command except serve and clean takes (or produces) a pipeline_run_id. That ID is the only argument that crosses the trace-context boundary.
--tts mock must be a first-class option, not a hidden test flag. The charter requires mock TTS to work before Piper is required.
Avoid a single podcast-pipeline run-end-to-end mega-command for Phase 1. It hides the approval gates, which are the architecturally interesting part.
